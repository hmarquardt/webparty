# Browser Party Line

Browser Party Line is a GitHub Pages-hosted WebRTC mesh video/audio chat app for small rooms. The static frontend connects to a Cloudflare Worker + Durable Object WebSocket signaling backend. The backend coordinates room membership and signaling only; it does not handle audio or video media.

## Architecture

- GitHub Pages serves `index.html`.
- The frontend connects to a signaling URL such as `wss://browser-party-line-signaling.YOUR_SUBDOMAIN.workers.dev/room/<roomId>`.
- The Worker validates `/room/<roomId>` WebSocket upgrade requests.
- The Worker maps each room to a Durable Object with `idFromName(roomId)`.
- The Durable Object tracks participants and routes signaling messages.
- Each browser creates one `RTCPeerConnection` per other participant.
- Media flows peer-to-peer when possible.
- Optional TURN relays media only when direct peer-to-peer fails.

## 1. Create the GitHub repo

1. Create a new GitHub repository.
2. Add `index.html` to the repo root.
3. Commit and push.
4. In GitHub, open:
   - Settings
   - Pages
   - Build and deployment
   - Deploy from branch
   - Branch: `main`
   - Folder: `/root`
5. Visit:

```text
https://YOUR_GITHUB_USERNAME.github.io/YOUR_REPO_NAME/
```

## 2. Create the Cloudflare Worker signaling backend

Install Node.js if needed.

From the project root:

```sh
cd cloudflare-worker
npm install
npx wrangler login
npm run deploy
```

After deploy, note the Worker URL:

```text
https://browser-party-line-signaling.YOUR_SUBDOMAIN.workers.dev
```

The WebSocket base URL will be:

```text
wss://browser-party-line-signaling.YOUR_SUBDOMAIN.workers.dev
```

## 3. Configure the frontend

1. Open the GitHub Pages URL.
2. Go to Admin / Config.
3. Set Signaling WebSocket URL base to:

```text
wss://browser-party-line-signaling.YOUR_SUBDOMAIN.workers.dev
```

4. Save config.
5. Create random room.
6. Copy invite link.
7. Open the invite link in another browser or device to test.

## 4. Local testing

Run the Worker locally:

```sh
cd cloudflare-worker
npm install
npm run dev
```

Use the local Worker WebSocket URL if Wrangler exposes one. For example, if Wrangler serves HTTP on port `8787`, the local WebSocket base is usually:

```text
ws://localhost:8787
```

For the frontend, open `index.html` directly or serve with:

```sh
python3 -m http.server 8000
```

Visit:

```text
http://localhost:8000/
```

Camera and microphone APIs usually require HTTPS or localhost. GitHub Pages is HTTPS, so production testing should work there.

## 5. Optional TURN setup

STUN is free and helps peers discover network paths. TURN relays traffic when direct peer-to-peer fails. TURN costs bandwidth money.

TURN options include:

- Twilio Network Traversal
- Metered.ca TURN
- Self-hosted coturn on a VPS

Add TURN details in Admin / Config. Do not hardcode paid TURN credentials in a public repo. For a public deployment, use short-lived TURN credentials generated by a trusted backend.

## 6. Recommended caps

- Max room size: 4
- Video: 640x480
- Frame rate: 24
- Warn at 4 users
- Auto-disable video at 5 users
- Encourage audio-only mode for larger groups
- Consider an SFU for 7+ participants

## 7. Troubleshooting

- Camera permission denied: check browser site permissions and make sure the page is HTTPS or localhost.
- No remote video: confirm both users joined the same room, accepted permissions, and the Worker WebSocket is connected.
- WebSocket failed: the frontend config must use `wss://`, not `https://`, for the deployed Worker base.
- Room full: the Worker default cap is 4 participants. Change `SERVER_MAX_PARTICIPANTS` in `cloudflare-worker/worker.js` and redeploy if needed.
- Works on same Wi-Fi but not elsewhere: direct peer-to-peer NAT traversal may be failing.
- VPN or corporate network issues: some networks block peer-to-peer media paths.
- TURN needed: add a TURN server in Admin / Config when direct connections fail.
- Browser autoplay blocked: click the page or use the room controls so the browser can start playback.
- GitHub Pages URL wrong: verify Pages is deploying from the `main` branch and `/root` folder.
- Worker URL mismatch: use `https://...workers.dev` in a browser health check, but `wss://...workers.dev` in frontend config.

## 8. Limitations

- Mesh WebRTC does not scale.
- No SFU.
- No recording.
- No authentication.
- No room passwords.
- No moderation.
- Anyone with invite link can join.
- Debug logs may reveal network/session metadata.
- localStorage config is not secure.

## Budget notes

GitHub Pages hosting is typically free for public repositories. Cloudflare Workers and Durable Objects have free/paid plan limits that can change, so check Cloudflare pricing before public use. TURN bandwidth is usually the first meaningful media cost because relayed video can use a lot of traffic.

## Configuration

Frontend configuration is stored locally in the browser using `localStorage`. Use Admin / Config to edit:

- Signaling WebSocket URL base
- Max room size
- Video width, height, and frame rate
- Audio-only default
- Max outgoing peer connections
- Screen share, chat, and debug toggles
- Device preferences
- ICE servers JSON
- Optional TURN settings

Server room cap is configured in:

```js
const SERVER_MAX_PARTICIPANTS = 4;
```

Change it in `cloudflare-worker/worker.js`, then redeploy the Worker.

## Testing checklist

- `GET /health` on the Worker returns `{ "ok": true }`.
- Frontend Admin / Config has the deployed `wss://` Worker URL.
- One browser can create and join a room.
- A second browser/device can open the invite link and join automatically.
- Local and remote video tiles appear.
- Mute/unmute updates participant status.
- Camera on/off updates participant status and remote media.
- Screen share starts and stops where supported.
- Chat messages appear in both browsers.
- Copy invite link works.
- Leave room closes peer connections and media tracks.
- Room cap returns a friendly room-full message.
- Debug tab shows signaling events, peer states, tracks, and stats.
